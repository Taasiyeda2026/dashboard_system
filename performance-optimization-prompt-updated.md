# פרומפט לאופטימיזציית ביצועים — Dashboard Taasiyeda

## הקשר
המערכת בנויה על GitHub Pages עם HTML/JS ES Modules, יחד עם Google Apps Script כ־backend ו־Google Sheets כמקור הנתונים.

הבעיה המרכזית היא חוויית ביצועים איטית במספר נקודות קריטיות:
- טעינה ראשונית איטית
- מעבר איטי בין מסכים
- פתיחת פירוט פעילות איטית
- תחושה כללית של המתנה מיותרת אצל משתמשים חוזרים

המטרה היא לבצע אופטימיזציית ביצועים ממוקדת, שמרנית ומעשית, בלי לשנות לוגיקה עסקית, בלי לשנות הרשאות, בלי לשנות מבנה נתונים, ובלי לשבור API קיים.

---

## עקרונות עבודה מחייבים

### לא לשנות
- לא לשנות action names ב־API
- לא לשנות response shape קיים, אלא אם מצוין במפורש ומגובה בצורך מדיד
- לא לשנות backend logic עסקי
- לא לשנות הרשאות
- לא לשנות חישובים
- לא לשנות פילטרים
- לא לשנות מיון
- לא לשנות מבנה נתונים קיים
- לא לבצע refactor רחב שלא נדרש ישירות לביצועים
- לא להוסיף ספריות חדשות
- לא לבצע redesign מלא של מסכים

### כן לבצע
- כן לבצע שיפורים ממוקדים בביצועים
- כן להשתמש במנגנוני cache קיימים ולהרחיב אותם בזהירות
- כן לבצע שיפורי perceived performance
- כן למדוד לפני ואחרי בכל שינוי משמעותי
- כן לשמור על backward compatibility מלאה
- כן לעבוד בצעדים קטנים, בטוחים, וניתנים לבדיקה

### מדיניות ביצוע
אם יש התלבטות בין פתרון אגרסיבי לפתרון שמרני — לבחור בשמרני.
אם יש ספק האם שינוי מסוים עלול לגרום לנתונים stale או לתקלות ניווט — לא ליישם לפני בדיקה.

## סעיפים שכבר קיימים בריפו — לא לבצע שוב

הסעיפים הבאים כבר קיימים או טופלו במבנה הקיים, ולכן אין לבצע אותם מחדש:

1. אין להסיר שוב את `Date2-Date35` מתוך summary — זה כבר טופל.
2. אין להוסיף מחדש קריאה ל־`backgroundSyncBootstrap()` במסלול של instant restore — זה כבר קיים.
3. אין להמציא cache key חדש שאינו תואם למנגנון המפתחות הקיים במערכת.

כל שינוי cache או prefetch חייב להתבסס על אותו מנגנון key generation שבו המערכת כבר משתמשת בפועל.

# חלק א׳ — Backend (Apps Script)

## 1. Warmup Trigger — צמצום Cold Start

### הבעיה
Apps Script נכנס למצב שינה לאחר פרק זמן ללא שימוש, והקריאה הראשונה אחר כך גורמת לעיכוב מורגש.

### המשימה
להוסיף פונקציית warmup פשוטה ב־backend, שמטרתה היחידה היא לחמם את סביבת הסקריפט.

### דרישות
- שם מוצע: `keepWarm()`
- הפונקציה לא תבצע `doPost`
- הפונקציה לא תכתוב נתונים
- הפונקציה לא תיצור side effects
- הפונקציה תיגש רק למה שנדרש כדי לחמם את הסקריפט

### דוגמה עקרונית
```js
/**
 * Warmup function — run via time-driven trigger every 10 minutes.
 * Prevents cold start by keeping the Apps Script instance warm.
 */
function keepWarm() {
  try {
    getSpreadsheet_();
  } catch (e) {
    // warmup only
  }
}
```

### תיעוד נדרש
לעדכן גם קובץ תיעוד קצר ב־backend עם הוראות הגדרה ידניות:
- פתיחת פרויקט Apps Script
- מעבר ל־Triggers
- הוספת Trigger לפונקציה `keepWarm`
- Time-driven
- Every 10 minutes

### הבעיה

### המשימה

```js

```
במקום:

- לאחר השינוי יש לציין בהערת מימוש היכן נכון לבצע invalidate חד־פעמי לאחר deploy
- אין להוסיף invalidate אוטומטי אגרסיבי ללא הצדקה

# חלק ב׳ — Frontend Cache ו־Data Loading

## 4. שיפור פתיחת Detail של פעילות בלי לנפח את Summary

### הבעיה
פתיחת פירוט פעילות מבצעת קריאת detail לשרת, גם במקרים שבהם כבר יש מספיק נתונים כדי לפתוח את ה־drawer מיידית.

### המטרה
לשפר את חוויית פתיחת הפירוט בלי להגדיל משמעותית את payload של מסך הפעילויות.

### המשימה
במסך הפעילויות לבצע את השיפור הבא:

#### שלב ראשון — פתיחה מיידית
כאשר המשתמש פותח פירוט פעילות:
- לפתוח את ה־drawer מיידית מתוך נתוני ה־summary שכבר נמצאים בזיכרון
- להציג מיד את השדות הבסיסיים שכבר קיימים
- לא להמתין ל־detail response לפני פתיחת ה־drawer

#### שלב שני — טעינה שקטה ברקע
אם חסרים נתוני detail:
- לבצע קריאת detail ברקע
- כשהנתונים חוזרים, לעדכן את תוכן ה־drawer בלי לסגור אותו
- לשמור על חוויית מעבר חלקה

#### שלב שלישי — memoization / detail cache
- לשמור detail cache לפי מזהה פעילות
- לא לבצע שוב ושוב את אותה קריאת detail עבור אותה רשומה
- אם detail כבר קיים, להשתמש בו מיד

### מגבלות חשובות
- לא להוסיף `meeting_schedule` מלא לכל שורת summary כברירת מחדל
- לא לנפח response של `activities` בלי מדידה ברורה שמצדיקה זאת
- ברירת המחדל היא לשמור על summary קל ומהיר

### רק אם יש הצדקה מדידה
רק אם לאחר בדיקה אמיתית מתברר שהוספת שדה קטן ל־summary חוסכת באופן משמעותי קריאות detail בלי לנפח payload בצורה בעייתית — אפשר לשקול זאת בזהירות.

## 5. הגדלת TTL ב־localStorage Cache — בזהירות

### הבעיה
יש מקום לשיפור במהירות חזרה למסכים שכבר נטענו, אבל הגדלת TTL אגרסיבית מדי עלולה להוביל לנתונים stale.

### המשימה
לבחון ולהתאים את ה־TTL של cache מתמשך במסכים שונים, בצורה שמרנית.

### עקרונות מחייבים
- לא לבצע הגדלת TTL אגרסיבית באופן עיוור
- לא לגרום לכך שמשתמשים יראו נתונים ישנים מדי לאחר שינויים תפעוליים
- לא לפגוע בעדכניות במסכים רגישים
- להעדיף stale-while-revalidate עקבי יותר, במידת הצורך, במקום פשוט “TTL גדול יותר”
- לא לשנות את עקרון ה־user-specific cache keys

### הנחיה מעשית
- מסכים יציבים יותר יכולים לקבל TTL מעט ארוך יותר
- `dashboard` ו־`activities` לא צריכים לקבל TTL אגרסיבי בלי refresh ברור ברקע
- `week` ו־`month` אפשר לשקול בזהירות
- כל שינוי צריך להיות מדוד ולא רק תיאורטי

### מטרה
מעבר מהיר יותר בין מסכים, בלי להקריב את עדכניות הנתונים.

---

## 6. Prefetch שקט — רק עם Cache Keys אמיתיים

### הבעיה
Prefetch לא נכון עלול לטעון נתונים למפתח cache שהמערכת בכלל לא צורכת.

### המשימה
להוסיף prefetch שקט למסכים נפוצים לאחר טעינת dashboard, אבל רק אם הוא נכתב לאותם cache keys שהמערכת משתמשת בהם בפועל.

### דרישות
- לא להשתמש ב־`cacheKey = route` אם זה לא תואם למפתח האמיתי
- להשתמש באותו מנגנון key generation קיים
- אם צריך, לחלץ helper משותף במקום לשכפל לוגיקה
- prefetch צריך להיות שקט, בלי loading UI, בלי שינוי state מורגש למשתמש
- לא לדרוס cache קיים ותקין

### מסכים לשלב ראשון
- `activities`
- `week`

### לא לבצע כרגע
- לא להוסיף prefetch גורף לכל המסכים
- לא לבצע prefetch למסכים נדירים או כבדים במיוחד לפני שיש מדידה אמיתית

### מטרה
להאיץ מעבר מהדשבורד למסכים הנפוצים באמת, בלי להכניס באגים של cache miss.

---

## 7. Targeted Cache Invalidation במקום ניקוי מלא

### הבעיה
לאחר mutations, ניקוי כולל של כל ה־cache גורם לטעינות מיותרות במסכים שלא באמת הושפעו.

### המשימה
לשנות את מנגנון ה־invalidation כך שינקה רק את מה שבאמת קשור לפעולה שבוצעה.

### עקרונות
- לא כל save צריך לנקות dashboard, week, month, activities ו־exceptions יחד
- יש להבחין בין mutations כבדים למutations קלים

### הבחנה נדרשת
#### Mutations כבדים
כאלה שמשנים פעילות, יומן, נתוני רשימות או מסכי overview

#### Mutations קלים
כגון:
- הערות פרטיות
- הרשאות
- שינויים נקודתיים שאינם משנים את כל התמונה

### דרישות
- לא לשנות את `state.js` אם אין בכך צורך
- לבצע invalidation ממוקד בלבד
- לשמור על עקביות עם מנגנון ה־screenDataCache הקיים

### מטרה
פחות reloads מיותרים, במיוחד אחרי שמירה נקודתית.

# חלק ג׳ — שיפור ניווט וחוויית טעינה

## 8. Debounce לניווט מהיר במסכי זמן

### הבעיה
לחיצות מהירות על ניווט בשבוע/חודש עלולות לייצר רצף בקשות מיותר.

### המשימה
להוסיף debounce עדין למסכי:
- `week`
- `month`

### דרישות
- ה־debounce צריך להיות קצר ועדין, למשל 120–180ms
- לא לשנות את לוגיקת הניווט עצמה
- לא לשנות offset logic או month navigation logic
- רק למנוע רצף requests מיותר מלחיצות כפולות/מהירות

### מטרה
למנוע בקשות כפולות בלי לשנות את חוויית הניווט.
