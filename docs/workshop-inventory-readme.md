# README — חישוב מלאי סדנאות / ציוד ומלאי

מסמך זה מתעד את אופן חישוב **מלאי הסדנאות** במערכת `dashboard_system`, בלשונית:

```text
ניהול הפעילות > ציוד ומלאי > מלאי סדנאות
```

המסמך מיועד למנהלי מערכת, תפעול, וקודקס/מפתחים עתידיים, כדי שכל שינוי עתידי במלאי יתבצע בזהירות ולא ישבור את החישוב.

---

## 1. מטרת מודול מלאי סדנאות

מודול מלאי סדנאות נועד לענות על השאלה:

```text
האם יש מספיק תוצרים / ערכות / ציוד להפעלת הסדנאות המתוכננות בקיץ?
```

המערכת מחשבת לכל קבוצת מלאי:

- כמה מלאי קיים.
- כמה כבר נוצל בפועל.
- כמה עוד צפוי להידרש.
- מה היתרה הצפויה.
- האם יש חוסר במלאי.
- האם צריך להעביר מלאי מעידן / הילה / מדריך אחר.
- מה מצב המלאי לפי מדריך.

---

## 2. שורה תחתונה של החישוב

החישוב עובד כך:

```text
קבוצת מלאי / stock_group_key
↓
כל הסדנאות שמחוברות לאותה קבוצת מלאי
↓
כל הפעילויות בפועל בקיץ 2026 שמתאימות לאותה סדנה
↓
חישוב לפי participants_count
↓
השוואה למלאי שחולק / נשאר אצל עידן / הילה / מדריכים
```

כלומר:

```text
יתרה צפויה = כמות קיימת - ניצול בפועל - צפי נדרש
```

## 3. טבלאות ומקורות מידע

| מקור | תפקיד בחישוב | שדות מרכזיים |
|---|---|---|
| `activities` | פעילויות בפועל שצורכות מלאי | `activity_no`, `activity_name`, `status`, `participants_count`, `instructor_name`, `instructor_name_2`, תאריכים |
| `lists` / `activity_names` | קטלוג הסדנאות הרשמי | `activity_no`, `activity_name`, `type`, `activity_type`, `stock_group_key`, `stock_quantity`, `stock_group_name` |
| `lists` / `workshop_stock` | מקור נוסף לפריטי מלאי / עריכת מלאי | `label`, `value`, `stock_quantity` |
| `workshop_stock_distributions` | חלוקת מלאי בפועל למדריכים / מלאי עידן / מלאי הילה | `stock_group_key`, `instructor_name`, `quantity_received`, `period_start`, `period_end` |
| state/UI | מצב פתיחת פירוט בממשק | `expandedWorkshop` |

## 4. מהי קבוצת מלאי / `stock_group_key`

קבוצת מלאי היא המפתח שמחבר בין כמה סדנאות שצורכות את אותו תוצר או ציוד.

לדוגמה:

| `stock_group_key` | סדנאות מחוברות | משמעות |
|---|---|---|
| `gitara` | `030` הגיטרה הראשונה שלי + `038` מנגנים פיזיקה | שתי סדנאות צורכות את אותו מלאי |
| `kofet_kesem` | `024` קופת קסם ד׳–ו׳ + `029` קופת קסם – מדע או אשליה | אותו מלאי |
| `shaon_robot` | `031` שעון רובוט + `032` הנדסת זמן – שעון רובוט | אותו מלאי |
| ללא `stock_group_key` | נוצר אוטומטית לפי מספר פעילות | לדוגמה `001` נהפך ל־`activity_1` |

## 5. נרמול מפתחות מלאי

המערכת מנרמלת מפתחות מלאי כדי למנוע כפילויות.

דוגמאות:

```text
activity_001 → activity_1
001          → activity_1
activity_030 → activity_30
```

כלומר, אין להסתמך על אפסים מובילים כמבדיל בין קבוצות מלאי.

---

## 6. איך נבנה קטלוג הסדנאות

קטלוג הסדנאות נבנה בעיקר מתוך `lists` בקטגוריה `activity_names`.

סדנה נכנסת לקטלוג אם מתקיימים התנאים הבאים:

| תנאי | פירוט |
|---|---|
| `category = activity_names` | שורת קטלוג של שמות פעילויות |
| `type = workshop` או `activity_type = workshop` | רק סדנאות |
| לא לא־פעילה | `active` / `is_active` לא false |
| יש שם פעילות | אחרת אין מה להציג |
| לא מוחרגת | תמיר / עולם הביומימיקרי וכו׳ מוחרגים |

## 7. החרגות

המערכת לא כוללת בחישוב מלאי סדנאות רגיל:

- פעילויות תמיר.
- `עולם הביומימיקרי`.
- `עולם הביומימיקרי הקסום`.
- פעילויות שאינן סדנאות.
- פעילויות שנמחקו.
- פעילויות שלא נמצאות בטווח הקיץ הרלוונטי.

## 8. אילו פעילויות נכנסות לחישוב

פעילות מתוך `activities` נכנסת לחישוב מלאי אם היא עומדת בכל התנאים:

| תנאי | הסבר |
|---|---|
| סטטוס `פתוח` או `סגור` | רק פעילויות תפעוליות |
| לא נמחקה | סטטוס נמחק לא נכנס |
| לא תמיר | תמיר מוחרג |
| משויכת לסדנה רשמית | לפי קטלוג הסדנאות |
| חופפת לטווח הקיץ | לפי תאריכי הפעילות |
| בטווח מלאי קיץ | `2026-06-15` עד `2026-08-30` |

הטווח הקבוע במודול מלאי סדנאות:

```text
WORKSHOPS_SUMMER_FROM = 2026-06-15
WORKSHOPS_SUMMER_TO   = 2026-08-30
```

## 9. התאמת פעילות לסדנה רשמית

פעילות בפועל יכולה להשתייך לסדנה רשמית בכמה דרכים:

| שיטת התאמה | דוגמה |
|---|---|
| `activity_no` זהה | פעילות מס׳ `030` משויכת לסדנה מס׳ `030` |
| מספר מנורמל זהה | `030` ו־`30` נחשבים אותו דבר |
| `activity_no` יוצר אותו `stock_group_key` | `activity_030` → `activity_30` |
| לפעילות עצמה יש `stock_group_key` תואם | אם קיים |
| שם פעילות מנורמל זהה לשם הסדנה | fallback לפי שם |

ההתאמה לפי שם היא fallback בלבד. עדיף להסתמך על `activity_no` ו־`stock_group_key`.

---

## 10. חישוב מספר תוצרים נדרש לפעילות

המערכת מחשבת צריכת מלאי לפי `participants_count`.

| מצב | חישוב |
|---|---:|
| יש `participants_count` חיובי | משתמשים במספר המשתתפים |
| `participants_count = 0` | 0 |
| חסר `participants_count` | 0 |
| `participants_count` ריק / null | 0 |

חשוב מאוד:

```text
אין fallback של 25 משתתפים.
```

כלומר, אם חסר מספר משתתפים, הפעילות תחושב כ־0 ותוצג אזהרה.

יש במערכת קבוע ישן:

```text
WORKSHOP_ESTIMATE_PER_ACTIVITY = 25
```

אבל לפי הבדיקות, הוא אינו משמש כ־fallback לחישוב מלאי.

---

## 11. אזהרה על חסר משתתפים

אם פעילות חסרה `participants_count`, המערכת:

- מחשבת אותה כ־0.
- מסמנת ליד שם הסדנה סימן אזהרה.
- מציינת כמה פעילויות חסרות מספר משתתפים.

משמעות תפעולית:

```text
יתרה צפויה יכולה להיראות טובה מדי אם חסרים נתוני participants_count.
```

לכן לפני קבלת החלטה על הזמנת מלאי, יש לבדוק שאין פעילויות חסרות משתתפים.

---

## 12. פתוח / סגור לצורך מלאי

פעילות נחשבת “סגורה” לצורך ניצול מלאי אם:

| מצב | נחשב |
|---|---|
| `status = סגור` | סגור |
| התאריך האחרון של הפעילות כבר עבר | סגור בפועל |
| `status = פתוח` ותאריך עתידי | פתוח |
| `status = נמחק` | לא נכנס לחישוב |

כלומר, פעילות שעדיין כתובה כ־`פתוח`, אבל התאריך שלה כבר עבר, תיחשב לצורך מלאי כפעילות שנוצלה בפועל.

## 13. נוסחאות טבלת מלאי ראשית

| עמודה בממשק | שם פנימי | נוסחה / מקור |
|---|---|---|
| מס׳ סדנה | `workshopNoDisplay` | מספרי הסדנאות המחוברות לקבוצת המלאי |
| שם הסדנה | `workshopName` | שם הסדנה / שם קבוצת המלאי |
| כמות קיימת | `stockQuantity` | סך כל `quantity_received` לאותה קבוצת מלאי |
| ניצול בפועל | `usedQuantity` | סכום משתתפים בפעילויות סגורות / שהתקיימו |
| צפי נדרש | `requiredQuantity` | סכום משתתפים בפעילויות פתוחות עתידיות |
| יתרה צפויה | `expectedBalance` | כמות קיימת − ניצול בפועל − צפי נדרש |
| סטטוס | `status` | מחושב לפי היתרה והמצב אצל מדריכים |

הנוסחה המרכזית:

```text
expectedBalance = stockQuantity - usedQuantity - requiredQuantity
```

## 14. חישוב “כמות קיימת”

במלאי סדנאות, `stockQuantity` מחושב מתוך `workshop_stock_distributions`.

המערכת מסכמת:

```text
stockQuantity = סכום quantity_received לכל הרשומות באותה קבוצת מלאי
```

הסכום כולל:

| סוג רשומה | נכנס לכמות קיימת? |
|---|---|
| מלאי עידן | כן |
| מלאי הילה | כן |
| מדריך רגיל | כן |
| לא משויך | כן |
| שורה עם 0 | כן, אבל לא משפיעה מספרית |

## 15. חישוב מיקומי מלאי

בפירוט הנפתח לכל סדנה מופיע סיכום מיקום מלאי.

| שורה | חישוב |
|---|---|
| מלאי עידן | סכום `quantity_received` שבהן `instructor_name = מלאי עידן` |
| מלאי הילה | סכום `quantity_received` שבהן `instructor_name = מלאי הילה` |
| יתרת מלאי מדריכים | כל מה שחולק למדריכים שאינם עידן/הילה |
| סה״כ מלאי כולל | עידן + הילה + מדריכים |

## 16. פירוט לפי מדריך

כאשר פותחים סדנה, מוצגת טבלת מדריכים.

| עמודה | מקור / נוסחה |
|---|---|
| מדריך | שם המדריך |
| כמות קיימת | כמה מלאי קיבל המדריך |
| ניצול בפועל | משתתפים בפעילויות סגורות של המדריך |
| צפי נדרש | משתתפים בפעילויות פתוחות של המדריך |
| יתרה צפויה | כמות קיימת − ניצול בפועל − צפי נדרש |

נוסחה:

```text
יתרה צפויה אצל מדריך =
כמות קיימת אצל מדריך
− ניצול בפועל של המדריך
− צפי נדרש של המדריך
```

---

## 17. פעילות עם שני מדריכים

המערכת יודעת לקרוא גם מדריך ראשון וגם מדריך שני.

שדות אפשריים:

```text
instructor_name
instructor
guide_name
guide
instructor_name_2
instructor_2
guide_name_2
guide_2
```

המערכת מסירה כפילויות. אם אותו מדריך מופיע גם כמדריך ראשון וגם כמדריך שני, הוא נספר פעם אחת.

אבל:

```text
אם יש שני מדריכים שונים באותה פעילות,
אותה כמות משתתפים מיוחסת לכל מדריך בפירוט המדריכים.
```

כלומר, המערכת לא מחלקת את המשתתפים חצי־חצי בין מדריכים.

משמעות: פירוט המדריכים נועד לבדוק מצב מלאי אצל כל מדריך, ולא בהכרח לסכום חזרה אחד־לאחד לטבלה הראשית.

## 20. דוגמה חישובית

נתונים:

| נתון | ערך |
|---|---:|
| כמות קיימת | 110 |
| פעילות סגורה | 64 משתתפים |
| פעילות פתוחה | 49 משתתפים |

חישוב:

```text
ניצול בפועל = 64
צפי נדרש = 49
יתרה צפויה = 110 - 64 - 49 = -3
```

תוצאה:

```text
חסר מלאי
```

## 25. בדיקות תפעוליות מומלצות

לפני החלטה על הזמנה / העברה / ניוד מלאי:

1. לבדוק שאין פעילויות חסרות `participants_count`.
2. לבדוק שסדנאות עם אותו תוצר מחוברות לאותו `stock_group_key`.
3. לבדוק שמלאי עידן / מלאי הילה / מדריכים הוזן נכון ב־`workshop_stock_distributions`.
4. לבדוק שאין פעילות שנשארה פתוחה בטעות אחרי שהתקיימה.
5. לבדוק פעילויות עם שני מדריכים.
6. לבדוק שפעילות לא נכנסה בטעות בשם אחר מהקטלוג.
7. לבדוק שסדנאות תמיר / ביומימיקרי לא נכנסות למלאי הרגיל.

5. לא לשנות נוסחאות בלי אישור מפורש.
6. לא להחזיר fallback של 25 משתתפים.
7. לא לשנות `stock_group_key` בלי בדיקת השפעה.
8. לא לשנות את ההחרגות של תמיר / ביומימיקרי בלי אישור.
9. לא לשנות DB / SQL בלי אישור מפורש.
10. אם שינוי JS/CSS בוצע — לעדכן `CACHE_VERSION` ב־`frontend/sw.js` ולהריץ build.

## 28. סיכום מקצועי

המודל הנוכחי של מלאי סדנאות הוא:

```text
לא לפי מספר פעילויות × 25
אלא לפי participants_count בפועל
```

המודל מתבסס על:

```text
קטלוג סדנאות
+
קבוצות מלאי
+
פעילויות בפועל
+
חלוקות מלאי
```

הגורמים הקריטיים לאמינות החישוב:

1. `participants_count` מלא ונכון.
2. `stock_group_key` נכון.
3. חלוקות מלאי מעודכנות.
4. התאמה תקינה בין פעילות בפועל לבין קטלוג הסדנאות.
5. הבנה שפעילות שעברה תאריך נחשבת ניצול בפועל.
