במסך הצעות מחיר צריך לבצע תיקון מרוכז לפי האפיון הבא, בלי לבצע שינויי SQL. את שינויי ה־SQL אני מריץ ידנית.

מטרה:
לסדר את מודל הלקוחות, אנשי הקשר והפעילויות בהצעות מחיר.

1. לקוחות ואנשי קשר

הגדרה:
לקוח = הגוף שמקבל את הצעת המחיר.
איש קשר = האדם אצל הלקוח.

הלקוח יכול להיות:

* בית ספר
* רשות / מועצה / עירייה
* אחר

טבלת הלקוחות בפועל היא contacts_schools.

בשמירת הצעת מחיר חדשה:
אם הלקוח לא קיים באנשי הקשר, צריך ליצור אותו אוטומטית ב־contacts_schools.
אם הלקוח כבר קיים, להשתמש בו ולא ליצור כפילות.

לאחר יצירת/איתור הלקוח, לשמור את ההצעה עם contact_school_id שמפנה אל contacts_schools.id.

חשוב:

* לא לשמור את שם הלקוח בתוך contact_name.
* contact_name הוא רק שם האדם.
* client_name הוא שם הלקוח לתצוגה.
* client_type הוא סוג הלקוח: school / authority / other.
* אם הלקוח הוא בית ספר:
  client_type = school
  client_name = שם בית הספר
  school = שם בית הספר
  authority = שם הרשות אם קיימת
* אם הלקוח הוא רשות:
  client_type = authority
  client_name = שם הרשות
  authority = שם הרשות
  school = null או ריק
* אם הלקוח הוא אחר:
  client_type = other
  client_name = שם הלקוח

ביצירת הצעה חדשה:
קודם לקרוא ל־RPC:
ensure_contact_school_from_proposal

ואז לשמור את ההצעה עם contact_school_id שהוחזר.

2. בחירת סוג חוזה

להשאיר הפרדה ברורה בין:

* סוג חוזה / סוג הצעה:
  הצעה משולבת
  פעילויות קיץ
  שנה הבאה

לבין:

* סוג פעילות:
  קורסים / תוכניות חינוכיות
  סדנאות / STEM ומייקרים
  סיור / התנסות בתעשייה

סוג חוזה קובע את מבנה המסמך.
סוג פעילות קובע אילו מוצרים/פעילויות מוצגים לבחירה.

3. תמיר

פעילויות שמתחילות ב־"תמיר" לא צריכות להופיע ברשימה הנפתחת של הצעות מחיר.
הן כן נשארות קיימות בפעילויות/קטלוג.
כלומר במסך הצעות מחיר יש להציג רק פריטים שבהם is_active_for_proposals = true.

4. הגדרות כוללות ופירוט מוצרים

יש פעילויות שהן הגדרה כוללת, למשל:

* סדנאות STEM
* סדנאות חלל
* חדרי בריחה

כאשר המשתמש בוחר הגדרה כוללת, המערכת צריכה להציע שתי אפשרויות:
א. להשאיר כהגדרה כוללת בהצעת המחיר.
ב. לפרט את המוצרים/סדנאות שתחת ההגדרה.

אם בוחרים להשאיר כללי:
להוסיף להצעה את שורת האב בלבד.

אם בוחרים לפרט:
להציג את פריטי הבן לפי:
parent_pricing_key = pricing_key של האב
proposal_display_mode = bundle_child
is_active_for_proposals = true

הגדרה כוללת מזוהה לפי:
proposal_display_mode = bundle_parent
או is_bundle_parent = true

פריט רגיל מזוהה לפי:
proposal_display_mode = single

פריט בן מזוהה לפי:
proposal_display_mode = bundle_child

5. טעינת רשימת הפעילויות להצעות מחיר

במסך הצעות מחיר להציג רק:
is_active_for_proposals = true

ברשימה הראשית להציג:

* פריטים רגילים
* הגדרות כוללות

לא להציג ישירות פריטי בן, אלא רק כאשר בוחרים "לפרט".

6. שגיאות ובדיקות

בכל insert/update להציג error.message אמיתי אם הפעולה נכשלת.
אחרי שמירת הצעה חדשה:

* הלקוח צריך להופיע באנשי קשר.
* ההצעה צריכה להישמר עם contact_school_id.
* אם נבחרה הגדרה כוללת, היא צריכה להישמר בהתאם לבחירה: כללי או מפורט.
