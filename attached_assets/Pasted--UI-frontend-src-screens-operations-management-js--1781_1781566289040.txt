קובץ:
`frontend/src/screens/operations-management.js`

אם יש CSS:
לעדכן רק את ה־CSS הרלוונטי.

לא SQL.
לא לשנות Supabase.
לא לשנות נתונים.
לא לעשות refactor רחב.
לא לטפל במובייל.

## הבעיה

אחרי תיקוני הרוחב נוצר חלל לבן גדול מדי.

כנראה class / CSS של צמצום רוחב הטבלה משפיע גם על טבלת **סידור עבודה**, ולא רק על **כמויות סדנאות**.

## מה צריך להיות

### 1. טבלת סידור עבודה

טבלת **סידור עבודה** צריכה לנצל את רוחב אזור התוכן.

לא להשאיר חצי עמוד לבן.

כלומר:

* wrapper של טבלת סידור עבודה: `width: 100%`
* הטבלה עצמה: `width: 100%`
* העמודות יתחלקו בצורה צפופה
* לא להחיל עליה width של 60% / 65%

עמודות סידור עבודה:

* תאריך
* יום
* שעות
* רשות
* בית ספר / מסגרת
* מדריך
* כיתה
* פעילות

### 2. כמות סדנאות

רק בלשונית **כמויות סדנאות** מותר לצמצם רוחב.

אבל גם שם לא להשאיר card רחב עם שטח לבן.

כלומר:

* הכרטיס של כמויות סדנאות עצמו יהיה צר
* הטבלה תמלא את הכרטיס
* לא מעטפת ברוחב מלא עם טבלה צרה בפנים

### 3. הפרדת CSS

לא להשתמש ב־class כללי שמשפיע על כל הטבלאות.

צריך class ייעודי לכל טבלה:

לדוגמה:

* `ds-ops-schedule-table-wrap` לסידור עבודה — רוחב 100%
* `ds-ops-workshops-card` לכמויות סדנאות — רוחב מוגבל
* `ds-ops-schools-view` לפי בתי ספר — קומפקטי אבל לא חצי עמוד ריק

## CSS רצוי

```css
.ds-ops-schedule-table-wrap,
.ds-ops-schedule-table-wrap .ds-table-wrap,
.ds-ops-schedule-table-wrap table {
  width: 100%;
}

.ds-ops-workshops-card {
  width: min(900px, 65%);
  margin-inline-start: auto;
  margin-inline-end: 0;
}

.ds-ops-workshops-card table {
  width: 100%;
}
```

## חשוב

* לא להשאיר חלל לבן גדול בסידור עבודה.
* לא להחיל צמצום רוחב גלובלי על כל הטבלאות.
* כמויות סדנאות צרה — כן.
* סידור עבודה רחבה ומנוצלת — כן.
* לפי בתי ספר קומפקטי — כן, אבל בלי כרטיסים ענקיים ובלי שטחים ריקים.

## בדיקות

1. לפתוח ניהול תפעול.
2. לבדוק סידור עבודה — הטבלה מנצלת את רוחב המסך ואין חצי עמוד לבן.
3. לבדוק כמויות סדנאות — הכרטיס עצמו צר, לא רק הטבלה.
4. לבדוק לפי בתי ספר — אין כרטיסים ענקיים ואין חללים לבנים גדולים.
5. להריץ `npm run check:build`.
6. לעדכן SW/cache אם צריך.

בסיום להחזיר:

1. איזה קבצים שונו.
2. האם צמצום הרוחב הוגבל רק לכמויות סדנאות.
3. האם סידור עבודה חזר לרוחב מלא.
4. האם החלל הלבן צומצם.
5. האם build עבר.
6. האם עודכן SW/cache.
