בקשה לעדכון Service Worker / Cache לכל המערכת

ריפו:
https://github.com/Taasiyeda2026/dashboard_system.git

נדרש לבצע עדכון Service Worker לכל המערכת, כדי לוודא שהמשתמשים מקבלים את הגרסה החדשה של הקבצים לאחר כל השינויים האחרונים במערכת.

המטרה:
למנוע מצב שבו הדפדפן מציג גרסה ישנה של המערכת בגלל cache / Service Worker / PWA / קבצים סטטיים ישנים.

יש לבדוק ולעדכן בזהירות את כל מנגנון ה־SW וה־cache במערכת.

דרישות:

1. עדכון גרסת Service Worker
- לעדכן את שם/מספר גרסת ה־cache.
- לוודא שכל cache ישן נמחק בזמן activate.
- להוסיף מנגנון ברור של cache busting לגרסה החדשה.

2. ניקוי קבצים ישנים
- לוודא שב־activate נמחקים cache names ישנים.
- לא להשאיר גרסאות ישנות של HTML / JS / CSS שממשיכות להיטען.
- לוודא שקבצי main.js, CSS, מסכים ו־assets נטענים בגרסה החדשה.

3. טעינה נכונה של עדכונים
- להוסיף/לוודא שימוש ב־skipWaiting אם זה תואם למבנה הקיים.
- להוסיף/לוודא שימוש ב־clients.claim אם זה תואם למבנה הקיים.
- לוודא שאחרי פריסה חדשה, רענון הדף מציג את הקוד החדש ולא גרסה ישנה.

4. אסטרטגיית cache
- עבור קובצי shell / html / js / css — לא לאפשר מצב שבו גרסה ישנה נתקעת.
- עבור assets סטטיים כמו לוגואים/תמונות — אפשר cache יציב, אבל עם גרסה מעודכנת.
- עבור קריאות Supabase/API — לא לשמור response ישן בצורה שמציגה מידע לא מעודכן.
- לא לגרום לכך שעמודי פעילויות, הרשאות, חריגות, ארכיון או הצעות והסכמים יציגו נתונים ישנים אחרי שמירה.

5. התאמה ל־PWA
- לבדוק את manifest אם קיים.
- לוודא שה־PWA עדיין נפתח תקין.
- לוודא שהמערכת לא נשברת במובייל או בדסקטופ.
- לבדוק התקנה / פתיחה כאפליקציה אם המערכת משתמשת בזה.

6. חשוב במיוחד
- לא לשנות לוגיקה עסקית.
- לא לשנות הרשאות.
- לא לשנות עיצוב מסכים.
- לא לשנות API.
- לא לשנות מבנה טבלאות Supabase.
- השינוי צריך להיות ממוקד ב־Service Worker, cache, versioning וטעינת קבצים עדכניים.

7. בדיקה של מצב קיים
- לבדוק האם SERVICE_WORKER_ENABLED מוגדר כ־false או true.
- אם ה־Service Worker כרגע כבוי, לא להפעיל אותו אוטומטית בלי לציין זאת בסיכום.
- אם נדרש להפעיל אותו כדי שה־PWA יעבוד, לציין במפורש מה השתנה ולמה.
- אם ה־SW כבוי בכוונה בגלל תקלות cache קודמות, לא להחזיר אותו בלי מנגנון בטוח לעדכונים וניקוי cache.

8. בדיקות נדרשות לאחר השינוי
- רענון רגיל מציג את הגרסה החדשה.
- Hard refresh מציג את הגרסה החדשה.
- פתיחה בחלון גלישה בסתר תקינה.
- לאחר deploy, לא נשארים כפתורים/מסכים ישנים מהגרסה הקודמת.
- עמוד פעילויות מציג את השינויים האחרונים.
- עמוד הרשאות נטען לאדמין.
- עמוד הצעות והסכמים נטען לפי ההרשאות.
- החיפוש בארכיון לא חוזר לגרסה הישנה.
- אין שגיאות console שקשורות ל־Service Worker או cache.

9. בסיום העבודה
נא לציין:
- אילו קבצים שונו.
- מה שם גרסת ה־cache החדשה.
- האם ה־Service Worker פעיל או כבוי.
- האם נמחקים caches ישנים.
- האם נדרש מהמשתמשים לבצע רענון / hard refresh / ניקוי cache חד־פעמי.
- אילו בדיקות בוצעו ומה התוצאה.

הדגש המרכזי:
כל המשתמשים צריכים לקבל את הגרסה העדכנית של המערכת ולא להיתקע על קבצים ישנים.
