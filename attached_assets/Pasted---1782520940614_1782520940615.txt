ריפליט, יש תקלה קריטית שחייבת להיפתר לפני כל המשך עיצוב:

במסך מדריך > אישורי ביצוע, כל לחיצה על כפתור:
"צפייה / הדפסה"
או כל כפתור צפייה באישור ביצוע

גורמת למסך לבן.

זה באג חוסם. לא להמשיך עיצוב ולא לבצע שינויים נוספים עד שהתקלה הזאת מתוקנת לחלוטין.

────────────────────────
1. מה התקלה
────────────────────────

כאשר מדריך לוחץ על כפתור צפייה / הדפסה באישור ביצוע, כל האפליקציה נופלת למסך לבן.

זה אומר שיש JavaScript exception לא מטופל, או שהמערכת מנסה לפתוח מסך/route/print view בלי הנתונים הנדרשים.

התקלה קיימת באזור מדריך בלבד.

────────────────────────
2. דרישת תיקון ראשית
────────────────────────

אסור שלחיצה על צפייה / הדפסה תפיל את האפליקציה למסך לבן.

בכל מצב:
- אם יש נתונים תקינים — לפתוח את אישור הביצוע לצפייה / הדפסה.
- אם חסרים נתונים — להציג הודעה ידידותית.
- אם אין הרשאה — להציג הודעה ידידותית.
- אם יש תקלה לא צפויה — להציג הודעה ולא להפיל את המסך.

הודעות אפשריות:
"לא ניתן לפתוח את אישור הביצוע. חסרים נתונים לפעילות זו."
או:
"אירעה תקלה בפתיחת אישור הביצוע. נסה לרענן את הדף או פנה למנהל המערכת."

────────────────────────
3. בדיקת שורש התקלה
────────────────────────

יש לשחזר את התקלה בפועל:

1. להתחבר כמדריך.
2. להיכנס ל"אישורי ביצוע".
3. ללחוץ על "צפייה / הדפסה".
4. לפתוח Console.
5. לזהות את השגיאה המדויקת.
6. לתקן את שורש השגיאה ולא רק להסתיר אותה.

לבדוק במיוחד:
- האם חסר approval object
- האם חסר activity object
- האם חסר row_id / activity ids
- האם חסרה רשימת activities
- האם הפונקציה מצפה לשדה שקיים בצד ניהול אבל לא קיים בצד מדריך
- האם יש undefined.map / undefined.forEach / Cannot read properties of undefined
- האם יש route שלא קיים למדריך
- האם יש הרשאת קריאה חסרה
- האם data-action לא מחובר נכון
- האם פתיחת ההדפסה משתמשת בנתונים מה־state הלא נכון

────────────────────────
4. לא ליצור מנגנון חדש כפול אם קיים מנגנון הדפסה
────────────────────────

אם כבר קיימת פונקציה קיימת להפקת / צפיית / הדפסת אישור ביצוע בצד ניהול או בצד מדריך, להשתמש בה.

לא ליצור מנגנון כפול אלא אם אין ברירה.

אם הפונקציה הקיימת דורשת התאמה למדריך, לבצע התאמה בטוחה:
- לוודא שהנתונים שמועברים אליה מלאים
- לוודא שיש fallback לשדות חסרים
- לוודא שהיא לא מניחה שהמשתמש הוא admin

────────────────────────
5. נתונים שחייבים לעבור לצפייה / הדפסה
────────────────────────

לפני פתיחת אישור ביצוע יש לבנות payload תקין הכולל לפחות:

- תאריך
- רשות אם קיימת
- בית ספר
- שם מדריך
- רשימת פעילויות
- שעות
- שכבות
- מספר משתתפים אם קיים
- row_id / activity ids
- סטטוס העלאה אם קיים

אם חלק מהנתונים חסרים, לא להפיל את האפליקציה.
להציג "לא זמין" או להשמיט שדה לא קריטי.

────────────────────────
6. הגנה מפני מסך לבן
────────────────────────

לעטוף את כל פעולת הצפייה / הדפסה ב־try/catch.

דוגמה עקרונית:

try {
  // build payload
  // validate payload
  // open print / preview
} catch (error) {
  console.error('[instructor approval print failed]', error);
  showToastOrAlert('לא ניתן לפתוח את אישור הביצוע. נסה שוב או פנה למנהל המערכת.');
}

בנוסף:
לא לאפשר exception לא מטופל בתוך render.
לא לשנות state למצב שגורם לרינדור של מסך ריק.
לא לנווט ל־route ריק או לא קיים.

────────────────────────
7. אם חסרים נתונים — לא להפיל
────────────────────────

אם אין מספיק נתונים לפתיחת אישור:
- לא לפתוח מסך לבן
- לא לזרוק exception
- להציג הודעה מסודרת
- להשאיר את המשתמש באותו מסך "אישורי ביצוע"

דוגמה:
"לא ניתן לפתוח את אישור הביצוע עבור תאריך זה כי חסרים נתוני פעילות."

────────────────────────
8. event handling
────────────────────────

לוודא שהכפתור "צפייה / הדפסה" מחובר נכון גם אחרי render דינמי.

אם הטבלה נוצרת ב־innerHTML, להשתמש ב־event delegation על container קבוע.

לוודא:
- data-action תקין
- data-key / data-id תקינים
- הכפתור יודע לאיזו שורת אישור הוא שייך
- אין stopPropagation שמונע פעולה
- אין preventDefault שגורם לתקלה
- אין overlay שחוסם קליקים

9. תיקון עיצובי באותו עמוד — אחרי תיקון המסך הלבן
────────────────────────

לאחר תיקון המסך הלבן, לבצע גם את ההתאמות שכבר נדרשו לעמוד אישורי ביצוע:

הטבלה החיצונית צריכה להיות קטנה וקומפקטית בלבד.

להציג רק:
תאריך | בית ספר | כמות פעילויות | אישור ביצוע | סטטוס | פעולה

לא להציג בטבלה החיצונית:
- רשות
- רשימת פעילויות מלאה
- שעות
- שכבות
- משתתפים
- מי איתי
- אחראי קשר
- טקסטים ארוכים

המידע המלא יופיע רק בתוך אישור הביצוע / צפייה / הדפסה / פירוט.

להקטין רוחב כולל:
max-width: 920px–980px;
margin-inline: auto;

להקטין פונטים:
td/th: 11px–12px;

להקטין כפתורים:
height: 24px–28px;
font-size: 11px;

להסיר כפילות סטטוסים:
עמודת "אישור ביצוע" = בחירת קובץ והעלאה.
עמודת "סטטוס" = תג אחד בלבד:
טרם הועלה / הועלה לבדיקה / אושר / נדחה.

לא להציג גם "לא הועלה" וגם "טרם הועלה" וגם "לא נבחר קובץ" בצורה כפולה.
